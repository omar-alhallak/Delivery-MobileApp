# Delivery App Clean Architecture
تعتمد Clean Architecture على تقسيم المشروع إلى طبقات رئيسية هي:
- Domain: يحتوي على الكيانات والقواعد الأساسية للنظام.
- Application: يحتوي على منطق التطبيق وحالات الاستخدام.
- Infrastructure: يحتوي على التعامل مع قواعد البيانات والخدمات الخارجية.
- API: الطبقة التي تتعامل مع المستخدم .
## 🌳 Branching
### Branch (`main`) : → فرع الإنتاج 
* A_ ❗ يمنع التعديل المباشر عليه.
* B_ ❗ يمنع إنشاء فروع منه.
### Branch (`Develop`) : → فرع التطوير الرئيسي 
* A_ ❗ يمنع التعديل المباشر عليه.
### Branch (`feature/*`) : → فروع الميزات

## ⚙️ طريقة العمل على البيئة:
1️⃣ أولاً: أقوم بإنشاء فرع feature/ من فرع Develop للميزة التي أرغب بإضافتها إلى Develop.

2️⃣ أقوم بعمل Puch من فرع feature/ الذي تم إنشاؤه ليظهر بال Git.

3️⃣ أبدأ بالتعديل أو إضافة أي شيء أريده أو أي عمل أقوم بتنفيذه.

4️⃣ عند الانتهاء من العمل، تكون طريقة الحفظ كالتالي:
   * أقوم بعمل Commit لحفظ التغيرات.
   * ومن ثم Puch لرفع العمل على Git.
    
5️⃣ إذا أردت أن تحدث برانشك تقوم بعمل Pull request عكسي الخطوات:
   * أقوم بعمل Merge لفرع Develop إلى feature/.
   * من فرعي feature/ أقوم بعمل Pull, ليتحدث الفرع.
     
6️⃣ عند الإنتهاء من العمل أقوم بعمل Pull request لفرع feature/ إلى Develop، وبذلك أكون قد انتهيت من feature/.

7️⃣ أقوم بعمل Pull لفرع Develop.


## 📋 قواعد العمل:
1️⃣ يمنع التعديل المباشر على فرعي main و develop منعاً باتاً.

2️⃣ يمنع إنشاء فروع من main منعاً باتاً.

3️⃣ يمنع قبول أي Pull Request، يجب إنشاء Pull Request فقط وأنا أقوم بمراجعتها وقبولها.

4️⃣ يمنع رفع أي كود غير مكتمل أو يحتوي على أخطاء.

5️⃣ كل فرع يجب أن يحتوي على ميزة واحدة فقط.

6️⃣ يجب كتابة شرح واضح عند إنشاء Pull Request لما تم العمل عليه مع إضافة Labels المناسبة.

7️⃣ يجب أن تكون تسمية الفروع واضحة مثل: 
```
* feature/login
* feature/order-system
* feature/driver-tracking
```
8️⃣ يمنع إضافة أو حذف أي مكتبة بدون موافقة.

9️⃣ يجب تنظيف الكود قبل رفعه من خلال:
  * حذف الـ using غير المستخدمة.
  * ترتيب الكود بشكل واضح ومنظم.
  * إضافة تعليقات مناسبة لشرح الأجزاء المهمة من الكود.
